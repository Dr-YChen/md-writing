# MD Writing

用于生成新的 Markdown 文档，或优化/润色已有文档的内容、结构和表达。

## 核心原则

### 表达方式

**关于括号的使用**：

英文缩写在 Markdown 文档中的使用规则：

1. **第一次出现**（仅限于正文）：必须使用 `中文名（英文全称, 缩写）` 格式
   - 示例：`时分复用（Time Domain Multiplexing, TDM）是一种……`
   - 下文即可直接使用"TDM"或"时分复用"
   - 注意：括号内先写英文全称，再写逗号和缩写——方便读者直接看到英文术语

2. **非第一次出现**：直接使用英文缩写或中文名称，无需额外说明

3. **标题中禁止使用括号注释格式**：标题只能使用中文，不得出现`中文名（英文全称, 缩写）`这种样式

**低频术语**：直接用中文，无需括号解释。

不要为了"解释"而加括号。括号是用来帮助读者识别即将反复遇到的缩写，不是用来给所有术语加备注的。

**不要自创比喻**：基于事实描述，避免使用个人创造的比喻来解释概念。直接陈述事实，比使用比喻更能准确传达意思。

**逻辑清晰**：使用清晰的文章结构，让读者能顺畅地从头读到尾。每个段落应有明确的主题句。

### 文档结构

```
# 标题

## 第一节
内容...

## 第二节
内容...
```

- 使用多级标题（`#`、`##`、`###`）构建层次
- 每一节聚焦一个主题
- 段落之间用空行分隔
- 避免过长的段落（超过 200 字考虑拆分成多段）

### 内容组织

1. **开头**：简要说明本文要讲什么，读者能获得什么
2. **正文**：按逻辑顺序展开（从概述到细节，或从原因到结果）
3. **收尾**：总结要点，必要时提供延伸阅读或下一步行动

### 概念解释规范（强制）

当文档中出现技术概念、理论术语或需要明确定义的抽象对象时，必须遵循以下顺序：

1. **定义优先**：首先给出概念的精确定义
2. **公式解释**：用数学公式表达核心定义（使用 `$$...$$` 行间公式）
3. **作用说明**：解释这个概念的具体作用、意义或应用场景
4. **深入讨论**：针对该概念展开相关问题讨论、局限性、扩展方向等

**示例结构**：

```markdown
## 卷积

卷积是两个函数在遍历过程中相互作用的数学运算。

**定义**：设 $f(t)$ 和 $g(t)$ 为两个连续函数，它们的卷积 $(f * g)(t)$ 定义为：

$$(f * g)(t) = \int_{-\infty}^{\infty} f(\tau) g(t - \tau) d\tau$$

**作用**：卷积用于描述一个信号在通过系统时的输出响应，广泛应用于信号处理、图像滤波和深度学习中的特征提取。

**讨论**：在实际应用中，卷积的计算复杂度较高，可以通过快速傅里叶变换（FFT）加速计算。
```

**注意事项**：
- 数学公式是概念定义的一部分，不是可有可无的装饰
- 如果概念没有公认的数学表达，可以用文字描述，但必须先给出精确定义
- 公式后的变量说明应紧跟其后，便于读者理解
- 讨论部分要有深度，不要停留在表面
- **变量符号格式**：如果概念有对应的变量符号，在首次定义时使用"中文名 + 变量符号"紧邻的格式，例如"信号带宽$B$"、"采样频率$f_s$"，后续正文直接使用变量符号，无需重复中文名

### 语言风格

- 使用主动语态
- 术语保持一致（同一概念全程用同一词汇）
- 名词和动词搭配协调
- 删除冗余的修饰词
- 数字的表述应符合上下文（阿拉伯数字 vs 中文数字）
- **翻译时一律中文化**：所有英文单词都应译为中文，不要直接留英文单词（如 resilience→韧性，legacy→遗留，reciprocal→互易的）。只有一种例外：第一次出现英文术语时在括号附上原文
- **列表慎用**：不要滥用列表格式（`-` 或 `1.`），AI 感很强。只有真正需要并列对比、步骤拆解、清单 Checklist 时才用列表；正常论述性内容用自然段落
- 优先用**连贯的句子和段落**表达逻辑流，列表是辅助工具不是默认格式
- **加粗慎用**：不要滥用 `**加粗**`，AI 感很强。只在需要强调重要概念或重要结论时使用加粗；正常叙述不要加粗，加粗不等于"重要"

### 写作检查清单

每完成一段后，自查以下问题：
- [ ] 括号是否必要？（只有高频缩写才加括号）
- [ ] 是否有直接留着的英文单词？（应全部中文化）
- [ ] 术语是否全文一致？（同一概念用同一词汇）
- [ ] 逻辑是否通顺？（读一遍看是否拗口）
- [ ] 数学公式是否用了正确的 `$$...$$`（行间）或 `$...$`（行内）？
- [ ] 图片语法是否在代码块外？
- [ ] 代码块是否仅用于展示代码本身？
- [ ] 列表是否必要？（能用段落讲清楚的就不用列表）
- [ ] 加粗是否必要？（能用正常叙述表达的就不加粗）

### Markdown 格式规范（强制）

**数学公式**：
- 行间公式（单独占一行）必须使用 `$$...$$`
- 行内公式（嵌入文字中）必须使用 `$...$`
- 禁止将公式放入代码块中

**图片**：
- 图片使用标准 Markdown 语法：`![alt](图片路径)`
- 不要将图片语法放入代码块中
- alt 文字应简洁描述图片内容

**代码块**：
- 仅在需要展示代码本身时使用代码块（`````` ``` ```````）
- 不要用代码块包裹：图片语法、数学公式、非代码内容

**标题层级**：
- 一个文档只用一个 `#` 一级标题
- 使用 `##` 和 `###` 构建层次结构
- 避免超过三级标题

**流程图与图表**：
- 如果文档中需要展示流程图、结构图、时序图等，使用 Mermaid 代码块
- 不要用 ASCII art（文本字符拼图）方式绘制图表
- Mermaid 代码块示例：

  ````markdown
  ```mermaid
  graph TD
      A[开始] --> B{判断}
      B -->|条件A| C[执行A]
      B -->|条件B| D[执行B]
  ```
  ````

- 支持的图表类型包括：`graph`/`flowchart`（流程图）、`sequenceDiagram`（时序图）、`gantt`（甘特图）、`pie`（饼图）等

**Python 绘图**：
- 如果需要精确控制图表样式（如配色、数据可视化、专业图形），用 Python 生成图片
- 在 Markdown 文件所在目录下新建 `code/` 文件夹存放绘图脚本
- 优先使用 `uv` 管理 Python 环境：检查 `uv --version` 是否可用，可用则用 `uv run` 执行，否则用系统默认 `python`
- 生成 PNG 图片到 `code/` 目录，用标准 Markdown 图片语法插入：`![](code/图片文件名.png)`
- 绘图库推荐：matplotlib、seaborn、plotly（生成静态 PNG 用 plotly.io.write_image）

### 翻译强制词表

以下词汇在翻译任务中**必须译为中文**，不得直接留英文：

| 英文 | 中文 |
|------|------|
| resilience / resilient | 韧性 / 抗扰性 |
| reciprocal | 互易的 / 倒易的 |
| legacy | 遗留的 |
| sub- / super- | 子- / 超- |
| coherent / incoherent | 相干的 / 非相干的 |
| localized / non-localized | 局部化的 / 非局部化的 |
| invariant / variant | 不变的 / 变化的 |
| separable / orthogonality | 可分离的 / 正交性 |
| perturbation | 扰动 |
| achievable | 可达的 |
| nominal | 标称的 |
| extrinsic / intrinsic | 外在的 / 内在的 |

如果不确定某个词是否应该翻译，就翻译它。

### 处理已有文档

如果用户提供了一篇文档让你优化：
1. 先通读全文，理解核心信息和结构
2. 检查表达问题（括号滥用、逻辑跳跃、冗余内容）
3. 修正问题，保持原有信息完整
4. 提升可读性和结构清晰度

### 处理翻译任务

如果要求翻译并保存为 Markdown：
1. 准确传达原文意思，不漏译
2. **一律中文化**：所有英文单词都译成中文，第一次出现时在括号附上英文原文（不是加括号解释，而是附原文）
3. 保持原文的段落结构和逻辑顺序

## 输出格式

输出应为纯 Markdown 格式，包含：
- 适当的标题层级
- 清晰的分段
- 简洁但完整的内容

如果输出过长（超过 500 行），考虑拆分成多篇文档，并在开头提供索引。
